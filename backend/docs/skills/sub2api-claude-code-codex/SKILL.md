# sub2api Claude Code Codex

This skill is the short entrypoint for running Claude Code through a local Headroom + `sub2api` Anthropic-compatible proxy chain backed by the user's OpenAI/Codex/ChatGPT subscription. Keep this file lean; load the reference files only when the task needs those details.

## Operating Rules

- Do not print OAuth tokens, API keys, refresh tokens, passwords, or copied auth files in chat.
- Do not wait for the user to catch persistence regressions. For every install, repair, Docker compose, Headroom, embedding-server, cache, Postgres, Redis, or "make reusable" task, proactively audit persistence before claiming done. The required proof is `docker inspect` showing `Type=bind` for Headroom `/root/.headroom`, `/root/.cache/headroom`, `/root/.cache/huggingface`, sub2api `/app/data`, Postgres parent `/var/lib/postgresql`, and Redis `/data`, plus `CCR_STORE True <nonzero>` after memory traffic. If any state path is a Docker named volume, fix the compose/profile/scripts first, recreate only the affected services, rerun `scripts/verify-claude-code-sub2api.ps1`, sync the local skill, commit, and push when repo changes were made.
- Prefer request-time `/v1/messages` probes and `usage_logs.upstream_model` over `/v1/models` catalog output.
- Keep Spark as Claude Code small-fast and compact first hop only for short/text compact work. Officially it is 128k and text-only, so do not route long 300k+ sessions or image-bearing compacts to Spark without the Luna fallback path. Set picker Opus/Fable/Sonnet defaults to Qwen high: `ANTHROPIC_DEFAULT_OPUS_MODEL=qwen3.8-max-preview`, `ANTHROPIC_DEFAULT_FABLE_MODEL=qwen3.8-max-preview`, and `ANTHROPIC_DEFAULT_SONNET_MODEL=qwen3.8-max-preview`, with display name `Qwen 3.8 Max`. Keep Haiku/subagent/small-fast choices evidence-driven: current summary bench says Spark remains the fastest compact/small-fast choice up to about 100k input and has no effort knob, Qwen3.8 Max is proven for near-1M high-effort summary, Qwen3.7+/Max, GLM-5.2, and DeepSeek-V4-Pro all passed a 225k-237k high-effort summary, while Qwen3.6 Flash and Spark failed around the 237k summary class. Do not promote GLM/DeepSeek to general subagent defaults from summary evidence alone because previous role/review probes showed empty or budget-sensitive visible output.
- Keep Luna as the second hop after Spark for small-fast/Haiku and compact fallbacks. Direct `gpt-5.6-luna` requests fall back to `gpt-5.3-codex-spark`.
- Keep normal message `model_fallbacks` empty unless the user explicitly asks for a fallback. Token Plan aliases must stay on the Alibaba account. Compact fallback is a separate compact-only contract.
- Treat Claude Code `count_tokens` as part of the mixed-provider contract. Qwen/GLM/DeepSeek Token Plan preflight must not hit the OpenAI/Codex account; patched sub2api routes `/v1/messages/count_tokens` through the same model-family classifier and returns a local estimate for Token Plan models. A 400 like `The 'qwen3.8-max-preview' model is not supported when using Codex with a ChatGPT account` means the running image or route is stale.
- Native `/compact` recognition is a two-proxy contract. Headroom must preserve the exact compact message, add `x-sub2api-claude-compact: 1`, and skip output shaping; sub2api treats that header as authoritative and applies `compact_model_mapping`. Spark has no configurable effort: patched sub2api must omit `reasoning.effort` even when the parent Sol/Terra request carries `max`, and Spark `usage_logs.reasoning_effort` must remain empty. For compact sessions containing image blocks, Spark's exact image-input HTTP 400 must activate the configured Luna fallback. Prove this with a real forked Claude Code `/compact`, Headroom's `headroom:claude_code_compact_prompt_preserved` marker, sub2api fallback logs, and `usage_logs`; a synthetic prompt alone is insufficient.
- For Claude Code `general-purpose`, `Explore`, and `workflow-subagent`, use user-level overrides at `%USERPROFILE%\.claude\agents\*.md` with `model: gpt-5.6-terra-medium` and `effort: medium` while Spark is quota-limited. The `-medium` alias is intentional: patched sub2api strips it to upstream `gpt-5.6-terra` and records `reasoning_effort=medium`, overriding inherited parent `max`. Normal message fallbacks stay empty unless the user explicitly enables them; a 0-visible-output Terra turn should surface as a debuggable proxy/upstream error, not silently switch to Sol. Project-specific agents such as `agent-marketplace-agent` should get the same frontmatter when they fan out heavily. Verify with agent JSONL and `usage_logs.reasoning_effort`, not with the Claude UI label alone.
- Do not persist `CLAUDE_CODE_EFFORT_LEVEL` in Windows User/System env or `~/.claude/settings.json env`. It overrides Claude Code's interactive `/effort` command for every session. Use `effortLevel` only as a soft startup default, and prefer `xhigh` for the profile default so users can switch to `max`, `high`, or lower efforts in-session.
- Keep `HEADROOM_OUTPUT_SHAPER=1` but force `HEADROOM_EFFORT_ROUTER=0`. Headroom's router defaults on when omitted and classifies clean `tool_result` turns as mechanical, rewriting an explicit Claude Code `output_config.effort=max` to `low`. Claude's JSONL still records the session effort as `max`, so UI/transcript proof alone is false-positive evidence. Verify a multi-turn tool loop with a pre-Headroom wire tap or equivalent request-body capture and match each turn to `usage_logs.reasoning_effort`.
- Do not install a global `PreToolUse` / `SubagentStart` / `SubagentStop` hook that blocks Agent calls unless the user explicitly asks for it. Current policy is advisory only: `workflowSizeGuideline=small` plus the `general-purpose` prompt asks for no more than 10 sibling agents and no deep chains. Do not claim Claude Code has a reliable built-in depth cap that prevents hundreds of descendants; local evidence has shown a single parent line can grow into hundreds of spawned agents.
- Never describe `400000`, `370000`, or `340000` as the GPT-5.6 upstream/model context limit. These are Claude Code client compact/display targets. Current safe client profile is `CLAUDE_CODE_MAX_CONTEXT_TOKENS=370000` and `CLAUDE_CODE_AUTO_COMPACT_WINDOW=340000`; still verify real upstream failures from proxy logs before blaming context.
- Treat `stale fake 429/503` as a proxy state bug first. For OpenAI OAuth 403, require adaptive `2s -> 15s -> 1m -> 5m` cooldowns, one bounded same-request retry after the first 403, and success cleanup that removes only the OAuth 403 temp state. For quota-origin `model_rate_limits`, a successful reset-credit call must clear the persisted limit and scheduler snapshot immediately; an external ChatGPT reset must be discovered by a same-model bounded reprobe after one minute, with a fresh upstream 429 closing that reprobe window again. Do not introduce a model fallback unless the user asks for one. A ten-minute first-hit cooldown, permanent OAuth disable, or manual DB reset as routine operation means the image is stale or regressed. Issue #1 separately covers quota-origin model cooldown recovery.
- Treat `429 {"detail":"Rate limited. Retry after ..."}` as Headroom's local token bucket, not an upstream subscription cooldown. Claude Code can expose it inside a tool result without replaying the lost call. The loopback profile must write `HEADROOM_RPM=6000` and `HEADROOM_TPM=100000000`; prove the effective `/stats.rate_limiter` values and run `scripts/test-headroom-rate-limit-burst.mjs` after every recreate. Do not accept source/env inspection alone.
- For Docker-in-WSL, verify both Headroom and sub2api health plus the Windows route. If Windows cannot reach `127.0.0.1:8787` but WSL/Docker can, publish Headroom on `0.0.0.0` and use the WSL eth0 IP on port `8787`; do not point Claude Code directly at sub2api `18081` except as a temporary diagnostic bypass. For a Hyper-V Claude host, configure `hyperv-bridge.env`; the single elevated autostart task must refresh the Default Switch -> current WSL `v4tov4` bridge, restrict its firewall rule to the current VM IP, update the VM settings over SSH, and prove health from that VM. A Docker health check from Windows is not sufficient.
- Keep `CLAUDE_CODE_DISABLE_NONSTREAMING_FALLBACK=1` in every Claude host, including native Linux/Hyper-V. Native compact remains intentionally non-streaming and is distinguished by the compact marker and Spark/Luna route. A normal Sol request with `stream=false`, no compact marker, multi-megabyte body growth, and repeated empty terminal responses is an emergency-fallback regression; trace it through `ops_error_logs` and the Claude transcript instead of mislabeling it as compact or adding an unsolicited model fallback.
- Treat `/bin/sh: node: not found` or `/bin/sh: python: not found` from `UserPromptSubmit` as a Claude-host hook runtime failure, separate from the inference gateway. On native Linux/Hyper-V, verify `node`, `python`/`python3`, the generated `~/.dev-pomogator/scripts/tsx-runner.js`, and its local `tsx` dependency from the same user and project before debugging Headroom. Prove the repair with a fresh `claude --print` process; an already-open process can retain the old hook registry and environment.
- Keep Headroom, lean-ctx, and TokenSave service/MCP binaries inside Docker for this profile. RTK is the deliberate exception: it must be installed in the OS where Claude Code executes Bash. Use `scripts/install-claude-rtk.ps1` for Windows plus WSL, or `scripts/install-claude-rtk.sh` for a native Ubuntu/Linux Claude host such as a Hyper-V VM. Keep exactly one global `PreToolUse(Bash)` hook, use `MSYS2_ARG_CONV_EXCL='*'` only on the Windows-to-WSL bridge, preserve the `cat`, `git diff`, `git show`, and `curl` accuracy exclusions, and bind the host RTK state to Headroom `/root/.local/share/rtk` when applicable. If `claude mcp list` shows stale host `headroom` or `tokensave` entries, remove them and re-add only Docker-backed Headroom.
- Windows PowerShell 5.1 must not pipe the synthetic RTK JSON payload directly into Git Bash. Its native stdin encoding can make `rtk hook claude` see empty or invalid JSON. Installer and verifier probes must use `Invoke-GitBashUtf8Stdin`, which base64-encodes UTF-8 in PowerShell and decodes it inside Git Bash before invoking the exact installed hook command.
- Keep `--embedding-server` enabled. The Headroom image must build from `stgmt/headroom` by `HEADROOM_GIT_REPO/HEADROOM_GIT_REF`; the local patch scripts remain as idempotent guardrails for overridden refs. If startup logs show `Falling back to per-worker embedder`, rebuild from the fork-aware Dockerfile and verify the ref labels.
- Treat `400 No tool output found for function call call_*` on a fresh Claude Code prompt as a Headroom mixed-memory continuation bug before blaming the VM or sub2api. The fork must defer client-owned calls when `memory_search` is emitted beside `Bash`/`Read`, replay only result-backed memory calls, and remove private memory definitions from the continuation. Prove it from the Claude JSONL, Headroom `Memory: Deferred ... client-owned tool call(s)` marker, and sub2api HTTP 200 continuation rather than by retrying the session.
- On NVIDIA hosts, set `HEADROOM_ACCELERATOR=cuda`, `HEADROOM_DOCKER_TARGET=gpu`, and `HEADROOM_KOMPRESS_BACKEND=pytorch`, and include `docker-compose.gpu.yml` for every build, start, and autostart. Do not claim GPU from host `nvidia-smi` alone: require non-empty Docker `DeviceRequests`, `torch.cuda.is_available()`, the expected device name, and `KompressCompressor().preload(...)=pytorch`. Use `benchmark-headroom-kompress` for CPU/GPU A/B; compare identical payloads, compression ratio, and sentinel retention.
- If WSL/Docker GPU access is blocked or Docker reports no CUDA `DeviceRequests`, do not leave CPU ONNX Kompress on the Claude Code hot path. Set `HEADROOM_FORCE_KOMPRESS=0`, `HEADROOM_DISABLE_KOMPRESS=1`, pass `HEADROOM_DISABLE_KOMPRESS` through compose, recreate Headroom, and verify `/health` reports `kompress.status=disabled` with `compression_executor.queued=0` and no leaked threads. Re-enable Kompress only after the GPU proof passes.
- Keep all service state on host bind mounts under `${SUB2API_STATE_ROOT:-./data}`. Headroom uses `headroom`, `headroom-cache`, and `headroom-huggingface`; sub2api uses `sub2api`; Postgres uses `postgres`; Redis uses `redis`. Do not replace these with Docker named volumes. Do not delete the state root unless the user explicitly wants to wipe memory, embeddings, accounts, database, and cache.
- When refreshing Codex/OpenAI OAuth after `Token revoked` / `account_disabled_auth_error`, verify both `status='active'` and `schedulable=true`. A recovered account with `schedulable=false` is still invisible to the scheduler and will surface as `no available accounts`, `503`, or Claude Code `Request timed out`. Current fork behavior restores `schedulable=true` from `ClearAccountError`, including the `apply-oauth-credentials` path.
- Keep Windows autostart single-owner. Remove or disable stale `headroom-proxy` scheduled tasks and Startup-folder `sub2api/headroom` `.cmd` launchers. A logon-only task is insufficient: the surviving task must have a one-minute repeating trigger, `RestartCount >= 3`, `MultipleInstances=IgnoreNew`, and call the health-first ensure script. The ensure script must not restart a healthy stack; after `ConnectionRefused`, WSL poweroff, Docker loss, or stale Hyper-V `portproxy`, it calls the full compose/bridge recovery and verifies both routes before succeeding. Normalize UTF-16/NUL WSL errors and retry transient `Wsl/Service/0x8007274c` failures on every WSL command, not only the initial Docker readiness probe.
- Keep the Headroom image bootstrap wrapper. `/usr/local/bin/start-headroom-proxy` seeds fresh persistent mounts from `/opt/headroom-seed` before starting `headroom proxy`, so a clean volume preserves memory without hiding bundled RTK/lean-ctx/difft/scc assets.
- Verify Headroom optimization with `docker exec headroom-sub2api headroom tools doctor`, `headroom savings --json`, and `headroom perf --format json`. A healthy stack should show bundled tools on PATH and nonzero proxy savings once traffic has passed through Headroom.
- After code changes, rebuild `sub2api-codex:local-token-usage` or the compose profile, recreate affected services under project `sub2api-codex`, then re-run live probes through Headroom.

## Minimum Done Criteria

For install/config/debug tasks, do not call it done until these are true or explicitly blocked:

- `headroom-sub2api` and `sub2api-codex` containers are healthy.
- Headroom `/health` reports ready and upstream `http://sub2api:8080`.
- Headroom image has `/usr/local/bin/start-headroom-proxy` and `/opt/headroom-seed` so fresh persistent volumes are seeded without overwriting existing memory.
- When `.env` selects CUDA, Docker exposes a GPU to `headroom-sub2api`, Torch reports CUDA and the expected device, Kompress preloads as `pytorch`, and `benchmark-headroom-kompress --require-cuda` completes with full sentinel retention.
- Headroom embedding-server logs show `Embedding server: ready.`, no per-worker fallback or missing watchdog module, `/tmp/headroom-embed-8787.sock` exists, and the memory factory returns `headroom.memory.adapters.watchdog SocketEmbedderClient 384`.
